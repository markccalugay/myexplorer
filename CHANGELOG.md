## [Unreleased]

### Added
- 2026-03-28: Replaced the React Native foundation screen with a planner-first mobile shell in `apps/mobile`, including manual trip naming, manual stop creation with lat/lng, stop reordering and removal, on-device draft/save persistence through AsyncStorage, reopen/resume flows, and an unsaved-change guard so the phone app is now useful for real pre-trip itinerary setup instead of serving only as a native placeholder. This materially advances issues `#48` and `#50`.
- 2026-03-27: Added `scripts/require-google-maps-config.mjs`, a new `npm run maps:doctor` workflow, and a focused maintainer note in `docs/google-maps-config-contract.md` so the repo now has a durable operational contract for Maps key setup instead of relying on changelog archaeology or a checked-in fallback key.
- 2026-03-27: Added a public-facing legal/footer surface to the web app with hash-addressable Privacy Policy, Terms of Service, Maps & Attribution, Permissions & Device Access, and Support pages, plus a new footer navigation component so the launch-ready legal routes now live inside the product instead of existing only as planning notes. This closes issues `#40`, `#44`, and `#46`, while materially advancing `#38` and `#45`.
- 2026-03-27: Added `src/lib/persistence.ts`, `src/lib/persistence.test.ts`, and `.env.example` so MyExplorer now has an explicit minimal-persistence layer for locally saved place/trip/navigation data and a real checked-in Maps env template to support fail-closed configuration.

- 2026-03-27: Added Android release-signing and runtime-config scaffolding in `apps/mobile/android`, including `apps/mobile/android/app/keystore.properties.example`, manifest metadata for a native Google Maps API key, and documented support for loading Maps config from `local.properties` or `MYEXPLORER_MAPS_API_KEY`.

- 2026-03-26: Added `src/lib/navigationSessionController.ts`, `src/lib/navigationSessionController.test.ts`, `src/lib/originSelection.ts`, and `src/lib/originSelection.test.ts` so live-origin lookup and persisted navigation-session restore/start/advance/end/sync behavior now live in shared modules instead of staying embedded inside `src/components/TripPlanner.tsx`.

- 2026-03-26: Added an initial `src/platform` adapter layer for storage, location, geocoding, and routing, then rewired the web app to start consuming those seams from `src/App.tsx`, `src/components/RoutePlanner.tsx`, and `src/components/TripPlanner.tsx` instead of reaching directly into browser globals and Google APIs in the same places.

### Changed
- 2026-03-28: Locked the next mobile routing direction around a backend-owned Google routing proxy, platform-native map rendering (`Google Maps` on Android and `Apple Maps` on iOS), persisted normalized route payloads for reopened trips, and automatic route refresh after structural trip edits. The active issue set now keeps `#49` as the native route-preview umbrella while tracking the newly separated backend proxy, persisted route payload, and automatic refresh work in `#55`, `#53`, and `#54`.
- 2026-03-27: Hardened the Google Maps env path so local web development now treats `.env.local` as the canonical source, `npm run dev` / `npm run build` fail early when the Maps key is missing, CI uses an explicit placeholder key for deterministic build verification, and the runtime error message now points developers to `npm run maps:doctor` plus the restart requirement after env changes. This closes issues `#51` and `#52`.
- 2026-03-27: Removed the reintroduced hardcoded Google Maps API key fallback from `src/lib/googleMapsConfig.ts`, restored the fail-closed Maps configuration path, and kept the environment setup consistent with the new `.env.example`, which closes issue `#2`.
- 2026-03-27: Tightened the web app's Google Maps compliance posture by limiting what gets written into `localStorage` for saved trips, saved places, and persisted navigation sessions, while also adding in-product attribution and disclosure copy around Google-derived content and location/storage usage. This closes issues `#39`, `#41`, and `#42`.
- 2026-03-27: Completed the Android shell scaffolding baseline by documenting and wiring the Android local config path, native Maps manifest metadata, local signing examples, and release-keystore flow in `apps/mobile/android`, which closes issue `#25` because the remaining Android work is now implementation on top of the shell rather than shell creation.
- 2026-03-27: Hardened the shared navigation session layer with validated session loading in `src/lib/navigationSessionStore.ts`, explicit restore outcomes plus reconnect/pause/resume/skip-stop controller helpers in `src/lib/navigationSessionController.ts`, and new regression coverage for those recovery flows. This work was necessary so future phone, Android Auto, and CarPlay surfaces can consume one stable recovery contract instead of duplicating session behavior in UI code, and it closes issue `#35`.
- 2026-03-27: Tightened the platform adapter boundary by shifting route, place, trip, geocoding, location, and session code onto a shared `GeoPoint` model. This adjustment was necessary because browser-only `google.maps` location types were still leaking across shared modules and making native reuse harder than it needed to be. With those contracts now provider-neutral by default, issue `#36` is closed.
- 2026-03-27: Added an in-car observability baseline to `AUTOMOTIVE_REFERENCE.md`, including lifecycle events, failure categories, minimum metadata, and guidance for keeping automotive telemetry distinct from generic Maps quota monitoring. This was necessary so later Android Auto and CarPlay work has an agreed debugging baseline before projection code gets deeper, and it closes issue `#31`.
- 2026-03-27: Expanded `AUTOMOTIVE_REFERENCE.md` from a high-level planning note into the canonical Android Auto and CarPlay execution baseline. It now defines Android Auto v1 scope, CarPlay v1 scope, phone-to-car handoff and reconnect behavior, cross-platform native permission and lifecycle requirements, and an automotive validation matrix, which closes planning issues `#28`, `#29`, `#30`, `#34`, and `#37`.
- 2026-03-27: Extended the shared navigation-session model with explicit pause, reconnect, reconnect-failure, resume, and skip-stop transitions, and added regression coverage for those flows so the session layer is a stronger foundation for future phone, Android Auto, and CarPlay recovery behavior.
- 2026-03-26: Advanced the iOS CarPlay foundation by adding a native `CarPlaySceneDelegate`, a placeholder `CarPlayManager` template flow, CarPlay-aware scene routing in `apps/mobile/ios/MyExplorerMobile/AppDelegate.swift`, and a scene manifest in `apps/mobile/ios/MyExplorerMobile/Info.plist`, so the mobile host is ready to attach a CarPlay scene once Apple approves the entitlement and the shared session wiring is finished.
- 2026-03-26: Fixed the first iOS/TestFlight archive gaps by wiring a real AppIcon asset catalog from `MyExplorer_Logo.png`, adding the required iPhone/iPad icon sizes plus `CFBundleIconName`, expanding iPhone/iPad supported orientations to satisfy iPad multitasking rules, and forcing Hermes to build from source with release `dwarf-with-dsym` symbols in the iOS Pod setup so future archives stop depending on a missing prebuilt Hermes dSYM.
- 2026-03-26: Hardened persisted trip-session restore by validating stored navigation sessions against the current trip fingerprint before resuming, preventing stale route sessions from silently rehydrating into `src/components/TripPlanner.tsx` after the underlying stop sequence changes.
- 2026-03-26: Chose and documented the mobile-native maps/routing strategy: Google Maps Platform remains the canonical route/place data source, the existing Google Maps JavaScript integration is now treated as the web adapter rather than the long-term shared contract, React Native phone screens are expected to use native-rendered maps, and Android Auto/CarPlay should project shared trip/session state into platform-native automotive surfaces. This closes issue `#33`.
- 2026-03-26: Tightened the React Native shell metadata and readiness docs by standardizing the native display name to `MyExplorer`, filling the iOS location permission copy, declaring Android location permissions for upcoming planner/navigation work, and updating the root/mobile readmes so the remaining release-signing, runtime-config, and shared-module integration gaps are explicit.
- 2026-03-26: Updated the automotive planning docs to point back to the new combined reference and advanced the planner implementation from documentation-only work into live foundation work for issues `#35` and `#36`. The current planner can now create, restore, sync, and clear a persisted navigation session while also routing more storage/location/geocoding/routing access through explicit platform adapters, even though the broader TripPlanner extraction and mobile-native adapter implementations are still open.
- 2026-03-26: Made the new React Native iOS setup repeatable in this repo by forcing source-based React Native pod installation in `apps/mobile/ios/Podfile`, avoiding the prebuilt pod artifact path failure caused by the current filesystem path containing spaces. The related native-shell issues were updated to reflect that iOS scaffolding is now in place and Android scaffolding exists, while Android verification and shared mobile integration work still remain.

- 2026-03-22: Stabilized map marker rendering so destination and pitstop pins no longer flash in and out during planner/map updates by keeping existing Google Maps markers alive when their identity has not changed instead of tearing down and recreating the full marker set on each render. In the same pass, auto-pitstops were converted from a softer suggestion threshold into a mandatory long-trip rule: trips over 50 km or over 60 minutes now require 1 pitstop, and trips over 100 km require 2 pitstops, with those stops distributed across the longest legs first so multi-stop trips still feel sensible.
- 2026-03-22: Removed the Trip Tools panel-count badge from the planner toolbox toggle, reworked the Convoy summary stats so labels sit above the numeric tiles for better fit, and added an inline Convoy help pop-up with hover/focus support on desktop plus tap-to-open behavior on mobile so travelers can quickly learn how to add vehicles, invite people, and review assignments without leaving the planner.
- 2026-03-22: Unified the Trip Tools experience across desktop and mobile by switching the web planner toolbox to the same tabbed panel flow used on mobile, re-theming the Convoy panel to the established blue MyExplorer surface language instead of the older peach palette, and fixing Route Recommendations label/title overflow so long names wrap cleanly in both the live recommendation card and the filter chip grid on web and mobile.
- 2026-03-22: Audited the recurring auto-pitstop instability against the recent planner history and fixed the underlying route-shape logic rather than the unrelated browser-extension noise seen from `lockdown-install.js`, `mf.js`, and “Node cannot be found in the current page.” The root issue was that auto-pitstop refreshes were effectively tied too loosely to the real manual trip shape, so changing manual middle stops could leave suggested gas stops stale, misplaced, or merged back into the trip in the wrong context. The planner now builds its pitstop refresh key from the full non-auto stop sequence, recomputes pitstops per real leg between manual stops, inserts suggested pitstops in leg order, dedupes repeat results more predictably, and assigns stable auto-pitstop IDs instead of random ones so equivalent Google Places results do not churn the UI unnecessarily. This keeps auto-pitstops aligned with the actual route users are planning and makes the behavior easier to reason about and test.
- 2026-03-22: Clarified and adjusted the Google Maps API-key recovery path after the March 20 fail-closed migration removed the checked-in fallback key. In theory the app should have loaded Maps from `VITE_GOOGLE_MAPS_API_KEY`, but in practice the running environment still reached the app without that value at runtime, so users kept seeing the app-shell banner that Maps services were unavailable and the key was missing. We first centralized Maps config lookup through shared helpers and added a Vite HTML meta fallback plus clearer guidance to restart or rebuild after env changes, but the live app still behaved as if no key had been injected. To restore reliability while that environment/runtime mismatch remains unresolved, the previous client-side fallback Google Maps key was reintroduced in the shared config resolver so Maps, Places, and routing can load again even when `VITE_GOOGLE_MAPS_API_KEY` is absent. This is an intentional temporary tradeoff: functionality is restored, but the client bundle once again contains the fallback Maps key until the runtime env delivery issue is fully fixed.
